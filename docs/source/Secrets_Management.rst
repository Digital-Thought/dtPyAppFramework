Secrets Management
==================

By default, the ``dtPyAppFramework`` library will automatically initialise a KeyVault secrets store for the user under which the application is running and also globally for any user that runs the application on the same computer systems.  These KeyVaults are referred to as the Application, User KeyVault and Application All Users KeyVault.

The Application User and All Users KeyVaults are based on ``pykeystore`` library developed by Marc Bertens-Nguyen.

In addition to the default Application User and All Users KeyVaults, the ``dtPyAppFramework`` library also supports the use of the AWS Secrets Manager and Azure KeyVaults.

Application User and All Users KeyVault Security
------------------------------------------------

Both the Application User and All Users KeyVaults are protected with a unique password.
The passwords are generated by the application and are unique for the instance of KeyVault.

Note: if you move the KeyVaults to a different location either on the same computer or to another computer, they will become inaccessible.

Defining AWS Secrets Manager and Azure KeyVault Access
------------------------------------------------------

As mentioned, in addition to the Application User and All Users KeyVault, the Secrets Manager also supports retrieving secrets from both the AWS Secrets Manager and Azure KeyVault.
It is possible to add multiple instances or either/both the AWS Secrets Manager and Azure KeyVaults to the Secrets Manager.

To add an AWS Secrets Manager and/or Azure KeyVault to the applications Secrets Manager, you must add the configuration to the ``config.yaml`` file for each the user or all users of the application.

The following is an example of an Azure KeyVault being added to the applications Secrets Manager:

.. code-block:: yaml

   secrets_manager:
     cloud_stores:
         my_secrets_store:
           store_type: Azure
           priority: 100
           azure_keyvault: name_of_key_vault 
           azure_identity_type: interactive_browser
           azure_tenant_id: 123


* ``secrets_manager.cloud_stores``\  is the section within the ``config.yaml`` where you can define your additional AWS and Azure stores.
* ``my_secrets_store`` is a unique name which you give to the store you are defining.
* ``store_type`` indicates the type of store you are defining.
* ``priority`` defines the precedence of the store and is used when retrieving secrets.

You can define multiple instances of cloud stores, however, the name of each cloud store must be unique:

.. code-block:: yaml

   secrets_manager:
     cloud_stores:

         my_secrets_store:
           store_type: Azure
           priority: 100
           Azure_keyvault: name_of_key_vault 
           Azure_identity_type: interactive_browser
           Azure_tenant_id: 123 

         my_second_secrets_store:
           store_type: Azure
           priority: 110
           Azure_keyvault: name_of_other_key_vault 
           Azure_identity_type: interactive_browser
           Azure_tenant_id: 890

**SafeGuarding Authentication Keys for Cloud Stores**

Depending on the method you choose to use to authenticate to either AWS or Azure, you will note that the coniguration requires some sensitive information in order to facilitate the authentication.

It is not recommended to place access keys, client secrets, passwords or other such information as plain text into the ``config.yaml`` file.  Instead, either the Application, User or All Users KeyVault can be used.

In the following example, we define an Azure KeyVault as a cloud store and use the ``key`` method for authentication but have not defined the required ``Azure_tenant_id``\ , ``Azure_client_id`` or ``client_secret`` values within the ``config.yaml``. 

.. code-block:: yaml

   secrets_manager:
     cloud_stores:

         my_secrets_store:
           store_type: Azure
           priority: 100
           Azure_keyvault: name_of_key_vault 
           Azure_identity_type: key

The value for the ``Azure_tenant_id``\ , ``Azure_client_id`` and ``client_secret`` will be retrieved from either the Application, User or All Users KeyVault (depending on which the value was stored to).
The secret key name under which the values be stored must be in the format of:


* ``secrets_manager.cloud_stores.my_secrets_store.Azure_tenant_id``\ ,
* ``secrets_manager.cloud_stores.my_secrets_store.Azure_client_id``\ , and
* ``secrets_manager.cloud_stores.my_secrets_store.client_secret``.

If the required values can not be found, then the connection to the specified cloud key store will fail.

*Refer to the instructions on adding a secret to the Applications KeyVault.*

Retrieving Secrets and Order of Precedence
------------------------------------------

When you are retrieving a secret via the ``dtPyAppFramework`` libraries Secrets Manager, the secrets stores are accessed in a defined order and the first occurrence of the requested secret found will be returned.

The following is the order of precedence:


#. Application, User KeyVault
#. Application, All Users KeyVault

Example of Precedence
^^^^^^^^^^^^^^^^^^^^^

As an example, say you have a secret named ``my.secret.key`` in the following KeyVaults with the indicated values:

.. list-table::
   :header-rows: 1

   * - KeyVault
     - Secret Value
   * - Application, User KeyVault
     - ``123456``
   * - Application, All Users KeyVault
     - ``7890``


If you request the value for the secret named ``my.secret.key`` it would return ``123456``. However, if the secret name of ``my.secret.key`` was not found within the Application, User KeyVault, it would have returned ``7890``.

Example of Precedence including Cloud Stores
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

When you define two or more cloud stores to be used within the Secrets Manager, the ``priority`` key defines the order of precedence the cloud store will fall within.

In the following example we have defined two cloud stores to be included into our Secrets Manager, one is for AWS and the other is Azure:

.. code-block:: yaml

   secrets_manager:
     cloud_stores:

         my_aws_secrets_store:
           store_type: aws
           priority: 100
           aws_profile: ec2 
           aws_region: ap-southeast-2

         my_Azure_secrets_store:
           store_type: Azure
           priority: 200
           Azure_keyvault: keyvault_name 
           Azure_identity_type: interactive_browser
           Azure_tenant_id: 123

Say you have a secret named ``my.secret.key`` in the following KeyVaults with the indicated values:

.. list-table::
   :header-rows: 1

   * - KeyVault
     - Secret Value
   * - my_aws_secrets_store
     - ``ABCDEF``
   * - my_Azure_secrets_store
     - ``QWERTY``


If you request the value for the secret named ``my.secret.key`` it would return ``ABCDEF``\ , as the priority of the ``my_aws_secrets_store`` is higher than the ``my_Azure_secrets_store`` store.

**NOTE: The Application, User and All Users KeyVaults take precedence over any defined cloud store.**
